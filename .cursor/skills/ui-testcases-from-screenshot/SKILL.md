# 根据截图生成测试用例

## 快速开始

当用户提供以下信息时：
- 一张或多张**页面截图**（可带标注），以及/或者
- **需求信息**（PRD 片段、用户故事、验收标准、业务规则、接口约束等）

从用户发起「生成测试用例」起，**一次性完成**：输出结构化测试用例并**同时生成**用例文档（.md）与 Excel 文件（.xlsx）。**仅保留 .md 和 .xlsx**，不保存中间 JSON 文件。

**生成用例时必须遵守「测试用例设计八大规则」**：等价类划分、边界值分析、场景法、错误推测法、因果图法、判定表法、正交试验法、状态迁移法。详见下方「测试用例设计八大规则」及 [reference.md](reference.md) 中的方法说明。

## 测试用例设计八大规则（必须遵守）

设计用例时，根据页面与需求**有选择地应用**以下八种方法，使用例既覆盖充分又不冗余：

| 方法 | 要点 | 在用例中的体现 |
|------|------|----------------|
| **等价类划分法** | 将输入域划分为有效/无效等价类，每类取代表值，减少重复 | 必填、格式、枚举：有效等价类取 1～2 条，无效等价类各取 1 条 |
| **边界值分析法** | 对范围、长度、数量取边界及边界外一档（如 min、max、min-1、max+1） | 长度 min/max、数值范围边界、分页首尾页、条数 0/1/满页 |
| **场景法（流程法）** | 按用户真实业务流程：主流程、分支、异常、回退 | 主流程一条完整路径；关键分支（如取消、返回、失败重试）各一条 |
| **错误推测法** | 凭经验补易错点：空、重复提交、权限越界、网络异常、并发 | 空值、重复、无权限、断网/超时、重复点击提交等反例 |
| **因果图法** | 分析输入条件与输出/结果的因果与约束，避免漏掉逻辑组合 | 多条件组合（如“有 A 且无 B 时才允许 C”）拆成条件+结果用例 |
| **判定表法** | 多条件多动作时列判定表，每列一条规则，覆盖条件组合 | 多条件组合（如筛选条件、权限×操作）用判定表列全组合或典型组合 |
| **正交试验法** | 多因素多水平时用正交表选代表组合，用较少用例覆盖主要组合 | 多维度组合（如 3 个下拉×多种取值）选正交代表，不穷举 |
| **状态迁移法** | 对有状态的对象画状态图，覆盖合法迁移与非法迁移 | 启用/禁用、待审/通过/驳回、草稿/已提交等状态及迁移路径 |

八大方法详细描述见 [reference.md](reference.md) 中「测试用例设计八大方法」。

## 工作流程（按顺序执行）

### 1）输入规范化

提取并简要复述：
- **产品/模块**与**页面名称**（用于填写用例表中的“模块”列）
- **目标端**：Web / iOS / Android / 小程序 / 桌面端
- **测试环境假设**：登录状态、角色权限、语言/地区、网络、开关/灰度等

若关键信息缺失，直接做**合理默认假设**，并在“假设”里列出来（不要反复追问）。

### 2）从截图理解 UI

对每张截图识别：
- **主要区域**：头部/导航/内容/底部、Tab、弹窗、抽屉等
- **可交互元素**：按钮、输入框、下拉、开关、链接、分页、查询与重置、筛选条件（多选/单选/模糊搜索）、表格行操作、行内编辑或备注入口等（以截图实际出现的为准）
- **页面状态**：空/加载/错误/成功、禁用、校验提示、Toast 等
- **文案**：可见标题、label、placeholder、错误提示（尽量原样引用）

若多张截图代表不同步骤/状态，请按时间顺序组织成一个**用户流程**。

### 3）把需求转成“可测规则”

将需求拆成：
- **业务规则**：约束、计算、权限、状态机
- **校验规则**：格式、范围、必填、唯一性（注意区分：同层级唯一 vs 全局唯一）
- **边界条件**：边界值、特殊字符、最大长度、并发/重复提交
- **非功能**：性能、可用性、可访问性、国际化、兼容性（仅在截图/需求明显相关时覆盖）

### 4）发散覆盖维度（按截图与需求决定，结合八大规则）

**原则**：覆盖范围由**当前页面的可交互元素**和**需求中的规则/约束**决定；设计时**结合测试用例设计八大规则**（等价类、边界值、场景法、错误推测、因果图、判定表、正交、状态迁移），在下列维度中有选择地应用，仅对截图或需求里实际存在的点设计用例：

- **主流程**：页面核心目标的一条完整成功路径（Happy path）
- **分支/替代路径**：多 Tab、多入口、多状态下的操作路径
- **校验**：需求或 UI 中出现的必填、格式、唯一性、范围、边界
- **权限**：需求若提到“按菜单/按钮控制”或“接口鉴权”，则覆盖功能权限与接口无权限时的表现
- **列表/表格能力**：若截图有筛选、查询、重置、分页、排序、行操作（编辑/删除/启用禁用等）、批量操作，则针对这些能力设计用例
- **二次确认与状态变更**：若有启用/禁用、删除等需确认的操作，覆盖单个/批量、确认/取消、状态校验与跳过规则
- **日志与审计**：若需求或 UI 有“操作日志”，则覆盖写入内容与展示方式（如按时间倒序）
- **行内编辑/备注**：若表格或详情有备注、行内编辑等入口，则覆盖编辑与保存后的展示
- **空态/加载/错误**：若 UI 或需求涉及无数据、加载中、接口报错等，则覆盖对应表现

不要求上述每项都出现；没有的入口或需求未提及的规则不强行编造用例。

### 5）生成结构化测试用例

按下面模板输出用例。

**书写规则**：
- 每条用例只聚焦**一个明确目标**；**模块**列填当前功能模块名（如“科目类型维护”“合并科目维护”）。
- **ID 编码规则**：每个模块的用例 **ID 必须从 `TC-001` 开始**，使用 3 位数字递增（`TC-002`…），**不跳号、不重复**；即使按“新增/编辑/权限”等分组展示，也仍保持同一连续序列。
- 步骤必须**可操作/可观察**，避免“验证是否正常”这类空话。
- 预期必须**具体**（文案、组件状态、跳转、数据变化；必要时可写接口层面的可观察结果）。
- 优先**黑盒验证**；仅当需求依赖接口/埋点/风控等时才写到网络/API。
- 若**编辑**的保存校验与**新增**完全一致，可输出一条“编辑操作同新增一致”的用例（步骤与预期可简写或引用新增），再视需要补充“编辑不同状态成功”“编辑时唯一性校验”等具体用例。

**通用写法**：步骤与预期要**具体到当前页面**（控件名称、提示文案、状态名以截图和需求为准）。例如：若页面有查询/重置，则步骤写清本页的筛选条件与点击查询/重置，预期写本页列表的刷新与数据范围；若需求提到操作日志，则按需求描述写“写入内容”“展示顺序”等；若页面有备注编辑，则写点击本页的备注入口、输入、保存及列表展示结果。不套用固定模板，以截图与需求为准。

## 输出格式（尽量按此结构输出）

### 假设
- [ ] …

### 范围
- **范围内**：…
- **范围外**：…

### 测试用例

用表格输出以下列（**必须包含**：模块、ID、标题、优先级、类型、前置条件、步骤、测试数据、预期结果、备注/覆盖点）：

| 模块 | ID | 标题 | 优先级 | 类型 | 前置条件 | 步骤 | 测试数据 | 预期结果 | 备注/覆盖点 |
|------|-----|------|--------|------|----------|------|----------|----------|-------------|

- **模块**：功能模块名称（如 科目类型维护、合并科目维护）。
- **优先级**：P0 / P1 / P2。
- **类型**：功能 / 校验 / 权限 / UI / 兼容 / 可访问性 / 性能。
- **前置条件**：无则填“—”或“无”。
- **步骤**：编号 1. 2. 3. …，可多行时用换行或分号分隔。
- **测试数据**：关键入参或数据条件；无则填“—”。
- **备注/覆盖点**：对应需求点、主流程/分支/校验/权限等简短标签。

### 可选：需求追踪（当需求有编号时）

若需求有编号（如 R1/R2/AC1），补一段映射：
- 需求编号 → 用例 ID 列表

## 6）保存用例文档并生成 Excel（每次生成用例时自动执行）

完成步骤 1～5 并得到测试用例表后，**在同一轮对话中必须**：

1. **保存 Markdown 文档**  
   将「假设 + 范围 + 测试用例表」保存为 **「模块名-测试用例.md」**（如 `公司档案配置-测试用例.md`），放在当前项目或 skill 目录下。

2. **生成 Excel 文件（不落盘 JSON）**  
   - 用例数据**仅在当前对话中使用**，**不要**再创建或保存任何 `.json` 文件。  
   - 编写并执行一段 **Python 脚本**：在脚本内将已生成的用例表定义为 **list of dict**（键：模块、ID、标题、优先级、类型、前置条件、步骤、测试数据、预期结果、备注/覆盖点），使用 `openpyxl` 写入工作簿，第一行为表头，后续每行一条用例。  
   - 输出文件命名为 **「模块名-测试用例.xlsx」**，与 .md 同目录。  
   - 执行后告知用户 .md 与 .xlsx 的路径。  

**最终只保留两个文件**：`模块名-测试用例.md`、`模块名-测试用例.xlsx`。不保留任何中间 JSON。

**Excel 列顺序**（与用例表一致）：模块 | ID | 标题 | 优先级 | 类型 | 前置条件 | 步骤 | 测试数据 | 预期结果 | 备注/覆盖点  

**Excel 单元格换行要求**：  
- **步骤**与**预期结果**写入 Excel 时必须为“单元格内多行”，即：`1. ...`、`2. ...`、`3. ...` 每条一行；`1）...`、`2）...` 每条一行。  
- 写入时在文本中使用换行符 `\\n`，并对单元格开启 `wrap_text=True`。  

依赖：`pip install openpyxl`（若未安装则先安装）。

## 质量闸（输出前自检）

- **八大规则**：等价类（有效/无效）、边界值、场景（主流程+分支）、错误推测（易错点）等在该页面适用处已体现；有状态迁移或复杂条件组合时，已用状态迁移/判定表/因果图思路覆盖
- 页面主要目标的**主流程**不缺失
- 需求中写明的**校验规则**（必填、格式、唯一性、边界等）均有对应用例，且关键规则至少 1 条**反例**
- 截图与需求中**实际出现的交互与能力**均有对应覆盖，不遗漏、不强行添加不存在的场景
- 用例标题不重复且描述明确

## 自我进化（每次生成用例后）

每次根据需求与原型生成测试用例后，可**提炼本次覆盖的新维度或模式**，并写入 [reference.md](reference.md) 的「常见覆盖维度」或「截图解读清单」，便于后续同类需求复用：
- **新页面类型**：若本次为配置类、报表类、审批流类等，且 reference 中尚无对应条目，可新增一条简短说明（何时覆盖、典型用例点）。
- **新校验/交互模式**：如“仅剩一条不可删”“导出文件名=报表名+期间”“手动更新弹窗必填”等，可归纳为通用维度的一句话描述。
- **不修改**：若当前需求均已被既有维度覆盖，则无需改动 reference，保持文档简洁。
