# 根据原型图与需求文档生成功能测试用例

## 快速开始

当用户提供以下信息时：
- **原型图/设计稿**（截图、设计稿、交互稿等），以及/或者
- **需求文档**（PRD、功能说明、用户故事、验收标准、业务规则、接口文档等）

从用户发起「生成测试用例」起，**按以下顺序一次性完成**：

1. **需求深度分析**（步骤 2）—— 从 PRD + 原型中逐项提取业务规则、UI 元素、校验规则、状态与权限
2. **功能测试点盘点**（步骤 3）—— 将分析结果转化为可逐一核对的测试点清单，自检通过后再继续
3. **识别业务链路**（步骤 4）—— 如有跨模块流程，梳理端到端链路
4. **生成测试用例**（步骤 5～6）—— 基于测试点清单与链路，生成功能点用例 + 业务链路用例
5. **输出文件**（步骤 7）—— 保存 .md 文档 + 生成合并 .xlsx

**仅保留 .md 和 .xlsx**，不保存中间 JSON 文件。

生成的用例分为两类，**分别维护在两个文件中**，便于独立调整与更新：
- **功能点用例**：聚焦单个功能/模块的增删改查、校验、权限、状态等
- **业务链路用例**：跨模块端到端场景，追踪一条完整的业务线从起点到终点的所有节点

**生成用例时必须遵守「测试用例设计八大规则」**：等价类划分、边界值分析、场景法、错误推测法、因果图法、判定表法、正交试验法、状态迁移法。详见下方「测试用例设计八大规则」及 [reference.md](reference.md) 中的方法说明。

## 全面覆盖合同（Coverage Contract，强制）

> 目标是“详细且全面”，默认按完整模式生成；不允许只给主流程的精简输出。

### A. 规则覆盖强约束
- 每条需求规则编号（R/AC/自编）必须映射：`规则 -> 测试点 -> 用例ID`。
- 每条规则至少 1 条正向用例；关键规则（校验/权限/状态/流程门禁）至少 1 条反向用例。
- 任何未映射需求必须在“未覆盖原因”中显式声明，不允许静默遗漏。

### B. UI 覆盖强约束
- 每个可交互元素至少覆盖 1 条正常操作用例。
- 每个高风险元素（导入、批量、删除、提交、切换、审批）至少再覆盖 1 条异常或边界用例。
- 若元素存在禁用/隐藏/只读状态，必须有状态差异用例。

### C. 状态与权限强约束
- 有状态对象必须覆盖合法迁移和非法迁移拦截。
- 角色-权限矩阵中每个“不可操作（❌）”都必须有无权限拦截用例。
- 数据权限场景必须覆盖“可见范围正确 + 越权拦截”两类结果。

### D. 风险节点优先级约束
- 导入、批量、并发、跨模块数据同步、资金类、外部依赖类默认 P0/P1。
- 涉及人工介入或时效窗口时，必须补中断恢复用例。

### E. 最小详细基线（不得低于）
- 功能点用例：每个模块至少覆盖 `主流程 + 关键校验 + 权限 + 状态 + 异常` 五类。
- 业务链路用例（有跨模块流程时）：至少 `1 条主链路 + 2 条关键分支 + 1 条中断恢复`。
- 输出必须包含：规则清单、测试点清单、追踪矩阵、覆盖率统计。

## 测试用例设计八大规则（必须遵守）

设计用例时，根据原型与需求**有选择地应用**以下八种方法，使用例既覆盖充分又不冗余：

| 方法 | 要点 | 在功能点用例中的体现 | 在业务链路用例中的体现 |
|------|------|---------------------|-----------------------|
| **等价类划分法** | 将输入域划分为有效/无效等价类，每类取代表值，减少重复 | 必填、格式、枚举：有效等价类取 1～2 条，无效等价类各取 1 条 | 链路中某节点的输入组合：不同角色/不同数据类型的通路 |
| **边界值分析法** | 对范围、长度、数量取边界及边界外一档（如 min、max、min-1、max+1） | 长度 min/max、数值范围边界、分页首尾页、条数 0/1/满页 | 链路中状态切换的边界条件（如金额刚好达到审批阈值） |
| **场景法（流程法）** | 按用户真实业务流程：主流程、分支、异常、回退 | 单模块内一条完整成功路径；关键分支各一条 | **跨模块端到端主链路**；分支链路（不同条件下不同路径）；中断恢复链路；回退链路 |
| **错误推测法** | 凭经验补易错点：空、重复提交、权限越界、网络异常、并发 | 空值、重复、无权限、断网/超时、重复点击提交等反例 | 链路中途异常中断（如审批人离职、库存不足扣减失败）、数据回滚一致性 |
| **因果图法** | 分析输入条件与输出/结果的因果与约束，避免漏掉逻辑组合 | 多条件组合（如"有 A 且无 B 时才允许 C"）拆成条件+结果用例 | 链路上游条件决定下游节点是否出现（如订单金额决定是否触发二次审批） |
| **判定表法** | 多条件多动作时列判定表，每列一条规则，覆盖条件组合 | 多条件组合（如筛选条件、权限×操作）用判定表列全组合或典型组合 | 链路中多个决策点的条件组合（如不同角色×不同状态×不同金额的审批路径） |
| **正交试验法** | 多因素多水平时用正交表选代表组合，用较少用例覆盖主要组合 | 多维度组合（如 3 个下拉×多种取值）选正交代表，不穷举 | 多角色×多状态×多分支的链路组合选取正交代表 |
| **状态迁移法** | 对有状态的对象画状态图，覆盖合法迁移与非法迁移 | 启用/禁用、待审/通过/驳回、草稿/已提交等状态及迁移路径 | 链路中业务对象跨模块的状态流转（如订单：草稿→已提交→待审批→已通过→发货中→已签收→已完成） |

八大方法详细描述见 [reference.md](reference.md) 中「测试用例设计八大方法」。

## 工作流程（按顺序执行）

### 1）输入规范化

提取并简要复述：
- **产品/模块**与**页面/功能名称**（用于填写用例表中的"模块"列）
- **目标端**：Web / iOS / Android / 小程序 / 桌面端 / 后端接口
- **测试环境假设**：登录状态、角色权限、语言/地区、网络、开关/灰度等

若关键信息缺失，直接做**合理默认假设**，并在"假设"里列出来（不要反复追问）。

### 2）需求深度分析（必做，不可跳过）

> **这是生成高质量用例的根基。分析不充分 → 测试点遗漏 → 用例覆盖不全。** 从 PRD 与原型中逐项提取，形成结构化分析结果。以下每个子步骤都必须执行。

#### 2.1 功能模块与页面拆解

列出：
- **一级/二级功能模块**（如"用户管理"包含"用户列表""新增用户""编辑用户"）
- **每个模块对应的页面/弹窗/抽屉**
- **模块间的依赖关系与数据流向**

#### 2.2 业务规则逐条提取（逐条编号，可追溯）

对 PRD 和原型中的每条规则**逐条提取并编号**，确保不遗漏。每条规则回答：**什么条件下 → 触发什么约束/计算/行为 → 产生什么结果**。

| 提取维度 | 要回答的问题 | 示例 |
|----------|------------|------|
| **约束类** | 什么条件下允许/不允许什么操作？ | "已提交状态的明细不可删除" |
| **计算类** | 涉及什么公式？输入什么、输出什么？ | "月度精力和=各项目精力之和，普通员工必须=1" |
| **权限类** | 什么角色能做什么操作？数据权限范围？ | "部门经理仅可见本部门数据" |
| **状态类** | 有哪些状态？如何迁移？触发条件？ | "草稿→已提交（提交）→已通过/已驳回（审批）" |
| **流程类** | 操作路径、页面跳转、审批流节点与条件？ | "提交后流转至直属上级审批" |
| **数据类** | 数据来源、展示规则、排序、格式化？ | "列表默认按创建时间倒序排列" |

**规则编号要求**：
- PRD 有编号则沿用（R1/R2/AC1），PRD 无编号则自行按模块编号
- **每条规则必须可追溯**——后续每条用例的"备注/覆盖点"必须标注对应规则编号
- **隐性规则也要提取**：原型交互中隐含的规则（如"仅剩一条时删除按钮置灰"）即使 PRD 未写也要编号

#### 2.3 UI 交互元素逐页盘点（逐页，不遗漏）

对**每张原型图/每个页面**，以树形结构列出所有可交互元素：

```
页面：XXX管理列表
├── 按钮：[新增] [批量删除] [导出]
│   └── 状态：正常/禁用(无权限时)/加载中
├── 筛选区
│   ├── [关键词搜索] 文本输入，模糊搜索
│   ├── [状态下拉] 多选
│   ├── [日期范围] 日期选择器
│   ├── [查询] 触发筛选
│   └── [重置] 清空筛选条件
├── 表格
│   ├── [分页] 默认20条/页
│   ├── [排序] 按创建时间/更新时间
│   ├── [行选择] 复选框+全选
│   └── [行操作] 编辑/删除/查看详情
└── 页面状态：空态/加载态/错误态
```

**关键要求**：
- 每个可交互元素标注对应的需求规则编号
- 标注元素在不同条件下的状态（正常/禁用/隐藏/加载/错误）
- **引用原型中的实际文案**（标题、label、placeholder、错误提示），尽量原样引用

#### 2.4 校验规则清单

| 校验层级 | 覆盖内容 | 触发时机 |
|----------|----------|----------|
| **字段级** | 必填、格式、长度min/max、数值范围、唯一性（同层级/全局） | 失焦 / 提交 |
| **表单级** | 字段间联动校验（开始≤结束）、条件必填（选"其他"时备注必填） | 提交 |
| **业务级** | 跨表单/跨流程校验（同一月份不可重复提交） | 提交 / 后端 |

#### 2.5 状态与权限矩阵

**状态迁移图**（对有状态的对象逐一画出，覆盖合法迁移与非法迁移）：

```
草稿 ──提交──▶ 已提交 ──审批通过──▶ 已通过 ──后续操作──▶ 已完成
  ◀──撤回──              ◀──驳回──
```

标注每个迁移的：操作名称、执行角色、前置条件、目标状态。

**角色-权限矩阵**（若需求涉及多角色）：

| 操作 | 角色A | 角色B | 角色C |
|------|-------|-------|-------|
| 新增 | ✅ | ❌ | ✅ |
| 编辑 | ✅（仅自己的） | ❌ | ✅（全部） |
| 删除 | ✅（仅草稿） | ❌ | ✅（全部） |

### 3）功能测试点盘点（分析 → 用例的桥梁）

> **在写测试用例之前，先把测试点列清楚。这是防止遗漏的关键步骤，不可跳过。**

#### 3.1 盘点规则

对步骤 2 中提取的**每条需求规则**，至少输出：
- **1 个正向测试点**（规则满足，验证正确行为）
- **1 个反向测试点**（规则违反，验证拦截/报错）

对步骤 2 中盘点的**每个 UI 交互元素**，至少输出：
- **1 个正常操作测试点**
- **1 个异常/边界测试点**（如禁用态、空数据、超长输入、重复提交）

#### 3.2 输出格式：功能测试点清单

| 模块 | 需求编号 | 测试点描述 | 测试类型 | 对应八大规则 |
|------|----------|-----------|----------|-------------|
| 用户管理 | R1 | 必填项全部填写，新增成功 | 功能 | 场景法 |
| 用户管理 | R1 | 用户名为空，提交失败并提示"请输入用户名" | 校验 | 等价类划分 |
| 用户管理 | R2 | 用户名=已存在用户名，提交失败提示"用户名已存在" | 校验 | 错误推测法 |

**测试类型**：功能 / 校验 / 权限 / 流程 / UI / 兼容 / 可访问性 / 性能

#### 3.3 盘点自检（写用例前必须逐项确认）

- [ ] 每条需求规则是否至少对应 1 个正向 + 1 个反向测试点？
- [ ] 每个 UI 交互元素是否至少对应 1 个测试点？
- [ ] 每个校验规则是否有正向（通过）和反向（拦截）测试点？
- [ ] 状态机中每种合法迁移是否有覆盖？每种非法迁移是否有拦截测试点？
- [ ] 角色-权限矩阵中每个 ❌ 是否有"无权限操作"测试点？
- [ ] 是否有需求规则只有正向、缺少反向/异常测试点？**如有，立即补充**

**自检通过后**，再进入后续步骤生成用例。

### 4）识别业务链路（跨模块端到端场景）

**这是生成链路式用例的核心步骤。** 从需求文档与原型中梳理出完整的端到端业务场景。

**4.1 识别参与角色与入口**
- 列出所有参与业务的**用户角色**（如提交人、审批人、操作员、管理员等）
- 每个角色的**入口操作**是什么（如提交订单、审批、发货、签收等）

**4.2 绘制业务链路图并评估复杂度**

先按时间顺序列出业务节点：`起点 → 节点A[角色:操作] → 节点B[角色:操作] → … → 终点`
- 标注每个节点的**触发条件**（如"订单金额>10万时触发总经理审批"）
- 标注节点间的**数据/状态传递**（如"提交后订单状态='待审批'，通知审批人"）

然后对每条链路评估复杂度，决定设计深度：

| 维度 | 简单（1分） | 中等（2分） | 复杂（3分） |
|------|-----------|-----------|-----------|
| 节点数 | 1~3 个 | 4~6 个 | 7+ 个 |
| 涉及角色 | 1 个 | 2~3 个 | 4+ 个 |
| 分支数量 | 0~1 个决策点 | 2~3 个决策点 | 4+ 个 |
| 外部依赖 | 无外部系统 | 1 个外部系统 | 2+ 个 |
| 逆向/回退 | 无回退路径 | 1 种回退场景 | 2+ 种回退场景 |

| 总得分 | 复杂度 | 用例数量参考 |
|--------|--------|-------------|
| 5~7 分 | 简单 | 1~3 条 |
| 8~11 分 | 中等 | 3~8 条 |
| 12~15 分 | 复杂 | 8~15 条 |

示例：
```
用户注册[用户] → 登录[用户] → 浏览商品[用户] → 加购[用户] 
→ 创建订单[用户] → 提交订单[用户] → 部门审批[部门经理]
→ (金额>10万?) 总经理审批[总经理] → 仓库发货[仓管员]
→ 确认收货[用户] → 订单完成[系统]
（评估：9节点/4角色/2决策点/2外部系统/2回退 → 14分，复杂链路）
```

**4.3 从链路中拆解出用例场景**

对每条识别出的链路，按以下维度拆解为链路测试用例：

| 链路类型 | 说明 | 用例要求 |
|----------|------|----------|
| **主链路（Happy Path）** | 从起点到终点的完整成功路径，所有节点按预期通过 | 至少 1 条，覆盖链路全节点 |
| **分支链路** | 不同条件下走不同路径（如金额阈值、角色差异、业务类型） | 每条关键分支 1 条 |
| **中断与恢复链路** | 链路中途某节点异常中断（驳回、撤回、库存不足、支付失败等） | 每种中断场景 1 条，含中断后的数据状态与恢复路径 |
| **回退链路** | 逆向操作路径（如取消订单、退货退款、审批撤回） | 每种回退场景 1 条，含各节点状态回滚校验 |
| **异常中断链路** | 系统异常导致链路中断（网络超时、服务宕机、数据不一致） | 关键中断场景至少 1 条 |

**4.4 识别链路中的关键决策点与风险节点**
- 条件分支：什么条件下走什么路径（用判定表/因果图梳理）
- 状态门槛：什么状态下允许/不允许什么操作（用状态迁移图梳理）
- 权限节点：什么角色在什么节点执行什么操作
- **风险节点识别**：对每个节点检查是否属于以下高风险类型，若是则对相关链路用例自动提升优先级至 P0，并增加异常覆盖：

| 风险类型 | 识别特征 | 用例增强动作 |
|----------|----------|-------------|
| 外部系统依赖 | 支付网关、第三方API、短信/邮件服务 | 增加超时、不可用、异常返回的链路用例 |
| 并发操作 | 库存扣减、秒杀、多人同时审批 | 增加并发冲突、数据一致性链路用例 |
| 超时/时效 | N分钟内支付、定时关单、Token过期 | 增加时间边界链路用例（刚好到期/过期后） |
| 金额/资金 | 支付、退款、结算 | 增加金额精度、负数、并发扣款链路用例 |
| 人工介入 | 人工审核、手动导入 | 增加人工节点遗漏/延迟的链路用例 |

详细风险节点清单见 [reference/coverage.md](reference/coverage.md) 中「风险节点识别清单」。

### 5）发散覆盖维度（基于测试点清单，结合八大规则）

**原则**：覆盖范围由**原型中的可交互元素**和**需求文档中的规则/约束**共同决定；设计时**结合测试用例设计八大规则**（等价类、边界值、场景法、错误推测、因果图、判定表、正交、状态迁移），在下列维度中有选择地应用，仅对原型或需求里实际存在的点设计用例：

**功能点维度**（单模块/单功能）：
- **主流程**：页面核心目标的一条完整成功路径（Happy path）
- **分支/替代路径**：多 Tab、多入口、多状态、多角色下的操作路径
- **校验**：需求或原型中出现的必填、格式、唯一性、范围、边界
- **权限**：需求若提到"按角色/菜单/按钮控制"或"接口鉴权"，则覆盖功能权限与接口无权限时的表现
- **列表/表格能力**：若原型有筛选、查询、重置、分页、排序、行操作、批量操作、导出等，则针对这些能力设计用例
- **表单能力**：若原型有新增/编辑表单、弹窗/抽屉、多步骤表单等，覆盖提交、保存、取消、草稿等
- **二次确认与状态变更**：若有启用/禁用、删除、提交、审批等需确认的操作，覆盖单个/批量、确认/取消、状态校验
- **消息与通知**：若需求提到通知、消息、提醒等，覆盖触发条件、内容、跳转
- **日志与审计**：若需求提到操作日志，则覆盖写入内容与展示方式
- **行内编辑/备注**：若原型中有备注、行内编辑等入口，则覆盖编辑与保存后的展示
- **空态/加载/错误**：若原型或需求涉及无数据、加载中、接口报错等，则覆盖对应表现
- **数据一致性**：若需求涉及多表关联、数据同步、缓存更新等，覆盖数据一致性场景

**业务链路维度**（跨模块端到端）：
- **端到端主链路**：覆盖从业务起点到终点的完整体验，验证跨模块数据流转与状态同步
- **分支链路**：不同条件（金额/类型/角色/渠道）下的不同业务路径
- **中断与恢复**：链路中途驳回/撤回/失败后的数据一致性、通知、及恢复后从断点继续或重新开始
- **回退链路**：逆向全流程（取消、退货、退款、撤回），逐节点校验状态回滚与关联数据恢复
- **跨角色协作**：多角色依次操作的完整协作流程，校验任务流转与权限衔接
- **定时/触发链路**：系统定时任务或事件触发驱动的自动化链路（如超时自动取消、自动确认收货）

不要求上述每项都出现；没有的入口或需求未提及的规则不强行编造用例。

### 6）生成结构化测试用例

用例输出分为两个部分：**功能点测试用例**与**业务链路测试用例**。

#### 书写规则（两类用例通用）

- 每条用例只聚焦**一个明确目标**
- 步骤必须**可操作/可观察**，避免"验证是否正常"这类空话
- 预期必须**具体**（文案、组件状态、跳转、数据变化；必要时可写接口层面的可观察结果）
- 优先**黑盒验证**；仅当需求依赖接口/埋点/风控等时才写到网络/API
- **每条用例必须关联需求**：在"备注/覆盖点"中标明对应的需求编号（如 R1、AC2）

#### 功能点测试用例

**书写规则**：
- **模块**列填当前功能模块名（如"用户管理""订单管理"）。
- **ID 编码规则**：每个模块的用例 **ID 必须从 `TC-001` 开始**，使用 3 位数字递增（`TC-002`…），**不跳号、不重复**；即使按"新增/编辑/权限"等分组展示，也仍保持同一连续序列。
- 若**编辑**的保存校验与**新增**完全一致，可输出一条"编辑操作同新增一致"的用例（步骤与预期可简写或引用新增），再视需要补充"编辑不同状态成功""编辑时唯一性校验"等具体用例。

#### 业务链路测试用例

**书写规则**：
- **链路ID**：统一使用 `E2E-001` 起编号，3 位数字递增，不跳号不重复。链路用例独立编号，不与功能点 TC 编号混用。
- **链路名称**：用 `→` 串联各节点，格式为"节点1 → 节点2 → 节点3"（如"用户注册→创建订单→审批通过→发货→收货"）
- **涉及模块**：列出该链路跨越的功能模块，按链路顺序排列
- **步骤**：每个步骤标注**所属模块**与**操作角色**，格式为 `[模块|角色] 操作描述`
- **预期结果**：每个步骤对应标注预期，格式为 `[步骤N] 预期描述`，确保每个节点的状态/数据变化可观察
- **测试数据**：链路级数据（如"订单金额=5万，角色=部门经理审批"），标注关键决策点的数据条件
- **备注/覆盖点**：标注链路类型（主链路/分支链路/中断恢复/回退链路）及相关需求编号

## 质量闸（输出前自检，分两阶段）

### 阶段一：分析完整性（在步骤 3 测试点盘点后自检）

- [ ] **规则全覆盖**：PRD 中每条业务规则、校验规则、流程规则均已提取并编号（含隐性规则）
- [ ] **原型全覆盖**：每张原型图的每个可交互元素均已盘点，无不明的"其他区域"
- [ ] **状态全覆盖**：有状态的对象均已画出状态迁移图（合法 + 非法迁移）
- [ ] **权限全覆盖**：有角色区分时已列出角色-权限矩阵
- [ ] **测试点不漏**：每条需求规则至少对应 1 正向 + 1 反向测试点，每个 UI 元素至少 1 条测试点
- [ ] **无纯正向规则**：检查是否存在只有正向测试点、缺反向/异常测试点的规则——如有，立即补充

### 阶段二：用例质量（在步骤 6 生成用例后自检）

- **八大规则**：等价类（有效/无效）、边界值、场景（主流程+分支）、错误推测（易错点）等在该功能适用处已体现；有状态迁移或复杂条件组合时，已用状态迁移/判定表/因果图思路覆盖
- **需求覆盖**：测试点清单中的每个测试点均有对应用例；关键规则至少 1 条**反例用例**
- **链路覆盖**：至少识别出 1 条主链路；存在条件分支时至少有分支链路；存在状态回退时至少有回退链路；存在审批/驳回时至少有中断恢复链路
- 功能主要目标的**主流程**不缺失
- 原型中**实际出现的交互与能力**均有对应覆盖，不遗漏、不强行添加不存在的场景
- **需求追踪矩阵**完整，每条需求编号均有对应用例 ID（功能点 + 链路）
- 用例标题不重复且描述明确
- 业务链路用例的步骤中，每个节点标注了 `[模块|角色]`

## 自我进化（每次生成用例后）

每次根据需求与原型生成功能测试用例后，可**提炼本次覆盖的新维度或模式**，并写入对应的 reference 子文件：
- **新功能模式**：写入 [reference/coverage.md](reference/coverage.md)「常见覆盖维度」
- **新校验/交互模式**：写入 [reference/validation.md](reference/validation.md)「常用校验目录」
- **新测试数据模式**：写入 [reference/test-data.md](reference/test-data.md)
- **新链路模式**：写入 [reference/chain-design.md](reference/chain-design.md)
- **不修改**：若当前需求均已被既有维度覆盖，则无需改动
