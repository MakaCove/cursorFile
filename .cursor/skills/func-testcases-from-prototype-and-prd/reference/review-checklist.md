# 用例评审清单模板

生成测试用例后，可基于此模板输出一份结构化的评审清单，供评审会议逐项核对。

## 使用方式

在技能输出用例后，按以下结构生成评审清单（可嵌入 .md 文档末尾或独立输出）。

---

## 评审清单格式

### 一、完整性检查

| # | 检查项 | 状态 |
|---|--------|------|
| 1.1 | 需求文档中每条功能规则是否有对应用例？ | ☐ |
| 1.2 | 需求文档中每条校验规则是否有对应用例（含反例）？ | ☐ |
| 1.3 | 每个模块的核心主流程是否有用例？ | ☐ |
| 1.4 | 原型中每个可交互元素是否有对应覆盖？ | ☐ |
| 1.5 | 需求追踪矩阵是否完整（需求编号→用例ID）？ | ☐ |

### 二、八大规则应用检查

| # | 检查项 | 状态 |
|---|--------|------|
| 2.1 | 等价类划分：有效/无效等价类均有代表值？ | ☐ |
| 2.2 | 边界值分析：min/max/min-1/max+1 是否覆盖？ | ☐ |
| 2.3 | 场景法：主流程+分支+异常是否完整？ | ☐ |
| 2.4 | 错误推测法：易错点（空值/重复/并发/权限）是否有反例？ | ☐ |
| 2.5 | 因果图/判定表：多条件组合是否梳理完整？ | ☐ |
| 2.6 | 状态迁移：合法迁移与非法迁移是否覆盖？ | ☐ |
| 2.7 | 正交试验法：多维度组合是否精简有效？ | ☐ |

### 三、用例质量检查

| # | 检查项 | 状态 |
|---|--------|------|
| 3.1 | 每条用例标题是否明确、不重复？ | ☐ |
| 3.2 | 步骤是否可操作/可观察（无"验证是否正常"）？ | ☐ |
| 3.3 | 预期结果是否具体（文案/状态/数据变化）？ | ☐ |
| 3.4 | 测试数据是否明确（无模糊的"任意值"）？ | ☐ |
| 3.5 | 优先级分配是否合理？ | ☐ |
| 3.6 | 备注/覆盖点是否标注了需求编号和覆盖类型？ | ☐ |

### 四、业务链路检查（如有链路用例）

| # | 检查项 | 状态 |
|---|--------|------|
| 4.1 | 是否至少识别了 1 条主链路？ | ☐ |
| 4.2 | 链路复杂度是否已评分？用例数量是否匹配？ | ☐ |
| 4.3 | 每条链路的步骤是否标注了 `[模块|角色]`？ | ☐ |
| 4.4 | 条件分支是否覆盖了所有关键决策点？ | ☐ |
| 4.5 | 中断/恢复链路是否覆盖了驳回、撤回、失败等场景？ | ☐ |
| 4.6 | 回退链路是否覆盖了取消、退货、退款等逆向操作？ | ☐ |
| 4.7 | 链路中的风险节点是否增加了异常覆盖？ | ☐ |
| 4.8 | 链路用例是否与功能点用例互补（无冗余）？ | ☐ |

### 五、模块维度检查

对每个功能模块逐项检查：

| 模块 | 新增 | 编辑 | 删除 | 查看 | 校验 | 权限 | 状态流转 | 异常 | 空态/加载 | 需求覆盖率 |
|------|------|------|------|------|------|------|----------|------|-----------|-----------|
| （模块名） | ☐ | ☐ | ☐ | ☐ | ☐ | ☐ | ☐ | ☐ | ☐ | _/N |

### 六、评审结论

| 项目 | 内容 |
|------|------|
| **总用例数** | 功能点 _ 条 + 业务链路 _ 条 = 共 _ 条 |
| **需求覆盖率** | _% ( _ / _ 条需求) |
| **P0 用例占比** | _% ( _ / _ 条) |
| **阻塞问题** | （列出评审中发现的必须修改的问题） |
| **建议优化** | （列出评审中建议的非阻塞优化点） |
| **评审结论** | ☐ 通过 / ☐ 修改后通过 / ☐ 不通过需重做 |

## 高风险专项评审（导入/切换/层级/提交流程）

| # | 检查项 | 状态 |
|---|--------|------|
| 7.1 | 导入是否覆盖：模板错误、数据错误、权限越权、部分成功、全失败？ | ☐ |
| 7.2 | 导入错误明细是否可定位到行号/字段并支持导出？ | ☐ |
| 7.3 | 层级删除是否验证父子级联、批量混合状态拦截、删除后汇总一致？ | ☐ |
| 7.4 | 视图切换是否覆盖前置校验、切换后数据/标记/状态继承一致？ | ☐ |
| 7.5 | 提交阻断类校验是否覆盖完整（必填、业务规则、状态门禁）？ | ☐ |
| 7.6 | 中断恢复是否覆盖（报错后修复重试）且无重复脏数据？ | ☐ |
| 7.7 | 并发/重复点击是否验证只生效一次？ | ☐ |
