# 业务链路识别与设计规范

## 什么是业务链路

业务链路是指一条完整的端到端业务场景，跨越多个功能模块，涉及多个角色的依次协作，从业务起点到终点的全过程。链路式测试用例验证的是**模块间的数据流转、状态同步与角色协作**，与功能点用例的"单点验证"形成互补。

---

## 链路复杂度评分

在识别出链路后，先对每条链路进行复杂度评分，据此决定用例设计的深度。

### 评分维度

| 维度 | 简单（1分） | 中等（2分） | 复杂（3分） |
|------|-----------|-----------|-----------|
| **节点数** | 1~3 个节点 | 4~6 个节点 | 7+ 个节点 |
| **涉及角色** | 1 个角色 | 2~3 个角色 | 4+ 个角色 |
| **分支数量** | 0~1 个决策点 | 2~3 个决策点 | 4+ 个决策点 |
| **外部依赖** | 无外部系统 | 1 个外部系统 | 2+ 个外部系统 |
| **逆向/回退** | 无回退路径 | 1 种回退场景 | 2+ 种回退场景 |

### 评分等级与设计深度

| 总得分 | 复杂度 | 链路用例最低要求 |
|--------|--------|-----------------|
| 5~7 分 | **简单** | 1 条主链路 + 有决策点时 1 条分支链路 |
| 8~11 分 | **中等** | 1 条主链路 + 所有分支链路 + 关键中断恢复（如驳回） |
| 12~15 分 | **复杂** | 1 条主链路 + 所有分支 + 所有中断恢复 + 所有回退 + 关键系统异常 |

用例数量参考范围：简单 1~3 条、中等 3~8 条、复杂 8~15 条。不机械追求数量上限，以实际覆盖为准。

## 链路识别流程

### 第1步：找业务起点与终点
从需求文档中找到每条业务线的：
- **起点**：业务由谁、在什么条件下触发（如用户提交申请、客户下单、系统定时任务触发）
- **终点**：业务的最终状态（如订单完成、审批结束、款项到账、数据归档）

### 第2步：识别中间节点与角色
按时间顺序理出每个操作节点：
- **谁**（角色）在**什么页面/入口**执行**什么操作**
- 操作后**产生什么数据/状态变化**
- 变化后**触发什么下游动作**（通知、任务流转、状态更新）

### 第3步：标注分支条件
识别链路中的决策点：
- 什么条件下走 A 路径，什么条件下走 B 路径
- 分支条件的数据来源（输入值、计算结果、配置项）
- 分支是否影响后续节点

### 第4步：标注状态门槛与风险节点
每个节点操作的前置状态条件：
- 什么状态下允许该操作
- 什么状态下禁止该操作
- 操作后的目标状态

同时识别每个节点的风险类型（外部依赖、并发、数据一致性等，详见 [coverage.md](coverage.md) 风险节点识别清单），对高风险节点额外增加异常覆盖。

## 链路分类与用例设计规范

### 1. 主链路（Happy Path）

**定义**：从起点到终点的完整成功路径，所有节点按预期推进，无中断、无驳回、无异常。

**设计规则**：
- 每个业务方向至少 1 条主链路用例
- 主链路用例必须覆盖所有必经节点，不可跳过
- 步骤中每个节点标注 `[模块|角色]`
- 预期中每个节点有对应的可观察结果

**示例场景**：
```
用户注册 → 登录 → 创建订单 → 提交审批 → 审批通过 → 发货 → 确认收货 → 完成
```

### 2. 分支链路

**定义**：在不同条件下，链路走不同的路径。分支由决策点决定。

**设计规则**：
- 每个关键决策点的每条分支路径至少 1 条用例
- 用例标题标明分支条件（如"金额>10万触发二次审批"）
- 测试数据中明确条件值

**常见决策点**：

| 决策点类型 | 示例 | 分支路径 |
|-----------|------|----------|
| 金额阈值 | 订单金额>10万 | 触发总经理审批 / 不触发 |
| 角色差异 | VIP用户 vs 普通用户 | 免审批 / 需审批 |
| 业务类型 | 采购入库 vs 销售出库 | 不同审批链 |
| 渠道差异 | 线上订单 vs 线下订单 | 不同发货流程 |
| 时间条件 | 工作日 vs 节假日 | 不同处理时效 |

### 3. 中断与恢复链路

**定义**：链路中途某节点被驳回、撤回、失败或异常中断，验证中断时的数据状态、通知、以及恢复路径。

**设计规则**：
- 每种中断场景至少 1 条用例
- 必须覆盖：中断触发 → 中断时的数据/状态校验 → 中断通知 → 恢复操作 → 恢复后继续流转
- 恢复路径可以是"从中断点继续"或"从头重新开始"，取决于需求

**常见中断场景**：

| 中断场景 | 示例 | 关键校验点 |
|----------|------|-----------|
| 审批驳回 | 部门经理驳回订单 | 驳回原因必填、状态=已驳回、提交人收到通知、订单可编辑后重提 |
| 用户撤回 | 提交人撤回待审批订单 | 状态回退到草稿、审批人待办列表移除该任务 |
| 库存不足 | 发货时库存不足 | 订单状态、是否自动拆分/等待补货/取消、通知客户 |
| 支付失败 | 提交订单后支付超时 | 订单状态（待支付/已取消）、库存释放、支付时效窗口 |
| 数据校验失败 | 下游系统数据校验不通过 | 错误信息返回、上游状态更新、人工介入入口 |

### 4. 回退链路

**定义**：逆向操作路径，从终态或中间态回退到前序状态，验证各节点状态回滚、关联数据恢复。

**设计规则**：
- 每种回退场景至少 1 条用例
- 必须覆盖：回退触发 → 逐节点状态回滚 → 关联数据恢复 → 回到起点或指定状态
- 关键校验：**数据一致性**——回退后所有关联模块的数据是否恢复一致

**常见回退场景**：

| 回退场景 | 示例 | 关键校验点 |
|----------|------|-----------|
| 取消订单 | 已支付订单取消 | 支付退款、库存回滚、积分回退、优惠券退还 |
| 退货退款 | 已签收后退货 | 退款金额计算、库存增加、财务凭证冲销 |
| 审批撤回 | 审批中撤回修改 | 审批记录保留、状态回退、重新提交后审批链是否从头开始 |

### 5. 异常中断链路

**定义**：系统异常（非业务）导致的链路中断，如网络超时、服务宕机、消息丢失。

**设计规则**：
- 关键外部依赖点至少 1 条异常用例
- 覆盖：异常发生时的用户提示、数据是否一致（无脏数据）、恢复后是否可继续

**常见异常场景**：
- 支付网关超时：订单状态不卡死、有超时关单机制
- 消息队列丢失：审批通知未送达时的补偿机制
- 第三方服务不可用：降级策略、重试机制
- 数据库死锁：并发操作时的数据一致性

## 链路用例与功能点用例的关系

| 维度 | 功能点用例 | 业务链路用例 |
|------|-----------|-------------|
| 覆盖范围 | 单模块/单功能 | 跨模块端到端 |
| 关注点 | 功能是否正确 | 流程是否通畅 |
| 步骤粒度 | 操作级（点击、输入、保存） | 节点级（提交、审批、发货） |
| 验证重点 | 单点UI/数据/提示 | 模块间数据流转/状态同步/角色协作 |
| ID 前缀 | TC-XXX | E2E-XXX |
| 是否可互推 | 不要求（已有功能点覆盖的节点在链路中可简化描述） | 可引用功能点，标注"详见 TC-XXX" |

**互补规则**：
- 链路用例中的每个节点，在功能点用例中应有对应的详细验证（如审批节点的必填校验、驳回原因格式等）
- 链路用例关注节点间的**流转**，功能点用例关注节点内的**细节**
- 如果某个节点在功能点用例中已充分覆盖，链路用例中可简化该节点步骤，标注"参见功能点 TC-XXX"

## 链路设计检查清单

输出链路用例前，对照检查：
- [ ] 是否识别了至少 1 条主链路？
- [ ] 是否对每条链路评定了复杂度？用例数量是否与复杂度匹配？
- [ ] 主链路是否覆盖了所有必经业务节点？
- [ ] 是否存在条件分支？若有，是否已为关键分支设计了分支链路？
- [ ] 是否存在审批/驳回/撤回？若有，是否已设计了中断恢复链路？
- [ ] 是否存在取消/退货/退款等逆向操作？若有，是否已设计了回退链路？
- [ ] 链路中是否识别了风险节点？高风险节点是否增加了异常覆盖？
- [ ] 链路用例的步骤中每个节点是否标注了 `[模块|角色]`？
- [ ] 链路用例的预期中每个节点是否有可观察的状态/数据变化？
- [ ] 链路用例是否与功能点用例形成了互补（无冗余、无遗漏）？
